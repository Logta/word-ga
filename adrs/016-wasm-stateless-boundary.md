# ADR-016: wasmを純粋関数として扱い、状態管理はTypeScript側に置く

**Status**: Accepted

## Context

MoonBit/wasm と TypeScript の責務をどう分担するかは、アーキテクチャ全体の保守性・パフォーマンス・テスト容易性に影響する。

選択肢として以下を検討した。

**案A: wasm ステートレス（現状）**
- wasm は `evolve(population, target, sel)` のような純粋関数のみをエクスポート
- 集団・世代数・履歴などの状態はすべて TypeScript の `SimState` が保持
- 毎世代、JS が集団を文字列にシリアライズして wasm に渡し、結果を受け取る

**案B: wasm ステートフル**
- wasm が集団を内部状態として保持
- JS は `init_population(target)` / `step()` / `get_population()` のような命令型 API を呼ぶ

## Decision

**案A（wasmステートレス）を採用する。**

## Consequences

### 案A を選ぶ理由

**シリアライズコストが無視できる**
集団サイズ30・最大染色体長100ビットで約3KB/ステップ。世代間隔300msに対してコストは無視できる。大規模化（集団サイズ数千・染色体長数万ビット）しない限り問題にならない。

**UIのために毎世代JSへ引き出す必要がある**
`IndividualList` / `GeneDisplay` など各コンポーネントが集団データを表示するため、wasmが状態を持っていても毎世代 JS 側へコピーする必要がある。ステートフルにしてもシリアライズを省けない。

**Vueリアクティビティとの相性**
wasm内部の状態変化はVueが検知できない。表示用にJSへコピーが必要になり、ステートフルにする利点が消える。

**MoonBitのモジュールレベルmut制約**
wasm-gcではモジュールレベルの `let mut` が使えず `Array` の可変性で代替が必要（ADR実装時に経験済み）。集団をwasm内部で持つと状態管理が複雑になる。

**純粋関数はテストしやすい**
入力と出力が明確で、hidden stateがないため単体テストが書きやすい。

### 責務の分担

| 責務 | 担当 |
|---|---|
| GA演算（交叉・突然変異・選択・適応度計算） | MoonBit/wasm |
| 状態管理（集団・世代数・履歴） | TypeScript (`SimState`) |
| 観測指標（多様性・統計値） | TypeScript（UIに必要なデータはJS側で集計） |
| UI・リアクティビティ | Vue + TypeScript |

### wasmをステートフルにする価値があるケース

- 世代を表示なしで一気に大量処理する「バッチ進化」モードを追加する場合
- 集団サイズや染色体長が桁違いに大きくなりシリアライズコストが支配的になる場合
- 多様性に応じて突然変異率を動的に変えるなど、観測指標をGA演算に組み込む場合
