# ADR-003: 適応度計算をビット単位で行う

**Status**: Accepted

## Context

遺伝的アルゴリズムで文字列を目標に近づけるとき、適応度（fitness）の計算方法として「文字単位の一致率」と「ビット単位の一致率」の2つが考えられる。文字単位では1文字一致かどうかの0/1評価になるが、ビット単位では部分的に近い文字（例: `A`=`00000` と `B`=`00001` は4ビット一致）も評価できる。

## Decision

適応度をビット単位で計算する方式を採用した。個体のバイナリ文字列と目標のバイナリ文字列を1文字（'0'/'1'）ずつ比較し、一致するビット数 / 総ビット数 を適応度とする。実装は `moonbit/src/individual.mbt` の `calc_fitness` 関数で行われる。

```
fitness = (一致ビット数) / (総ビット数)
```

## Consequences

### メリット
- 目標に近い文字を持つ個体が高い適応度を得られるため、進化の勾配がなめらかになる
- 文字単位評価に比べて選択圧が細かく働き、収束が速くなる傾向がある
- 実装がシンプル（`'0'`/`'1'` の文字比較のみ）

### トレードオフ
- ビット表現と文字の視覚的な対応関係が直感的でない場合がある

### Known Limitation: スペースのビット表現問題

5ビットエンコーディングでは、インデックス 27〜31（`11011`〜`11111`）がすべてスペース文字としてデコードされる（`CHARS[index] || " "` の挙動）。そのため、画面上では同じスペースに見える文字に複数のビット表現が存在する。

適応度計算はビット比較で行われるため、目標がスペースの場合（エンコード値 `11010`）、`11011`〜`11111` のビット列を持つ個体はスペースと表示されながらも適応度 1.0 を得られない。例えば `11111` はスペースと表示されるが、目標 `11010` との一致は5ビット中3ビットのみ（fitness = 0.6）となる。これはビット単位適応度と文字表示の不一致であり、設計上の既知の制限である。
